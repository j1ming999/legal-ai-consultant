## 1. 项目概述

### 1.1 项目背景

随着中国法律法规体系的持续完善（民法典2021年施行、个人信息保护法、数据安全法等陆续出台），公众对法律咨询的需求日益增长。但传统法律咨询服务面临三大困境：

- **成本高**：律师咨询费动辄数百上千元
- **资源分布不均**：优质法律资源集中在一二线城市
- **信息混乱**：网络上法律信息真伪难辨，普通民众难以甄别

大语言模型（LLM）虽然具备强大的自然语言理解和生成能力，但存在两个突出问题：

1. **知识幻觉**：模型可能生成看似合理但实际错误的法条内容
2. **知识截止**：模型训练数据有截止日期，无法反映法律的最新修订

### 1.2 项目目标

构建一个基于 RAG（检索增强生成）技术的法律智能问答系统，实现：

- 覆盖 180+ 部中国法律法规的知识库
- 用户用自然语言提问，系统检索相关法条并生成专业回答
- 回答可溯源——每条结论都能追溯到具体的法律名称和条文编号

### 1.3 核心创新点

1. **四级检索管道**：向量语义检索 → 关键词精确匹配 → 等权RRF融合 → LLM查询改写
2. **等权RRF融合策略**：通过实验验证等权配置优于不等权配置，关键词结果获得公平竞争机会
3. **阿拉伯数字-中文数字双向转换**：解决用户口语中"148条"与法律原文"第一百四十八条"的匹配问题
4. **装饰器模式设计**：QueryRewriteRetriever包装HybridContentRetriever，不改动原有检索代码即可增加改写能力

## 4. 核心技术：RAG 检索增强生成

### 4.1 RAG 原理

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大语言模型相结合的技术架构。其核心思想是：**在生成回答之前，先从外部知识库中检索相关信息，然后将检索到的内容作为上下文注入大语言模型，让模型基于实际材料作答，而非仅依赖模型内部参数化知识。**

```
标准 LLM 回答流程：
  用户提问 ──→ LLM 直接生成 ──→ 回答 (可能幻觉)

RAG 回答流程：
  用户提问 ──→ 检索知识库 ──→ Top-K 相关片段
       │                            │
       └──────── 拼接提示词 ─────────┘
                       │
                       ▼
                  LLM 基于材料生成 ──→ 回答 (可溯源)
```

### 4.2 RAG 相比纯 LLM 的三大优势

1. **可溯源**：每条结论都能对应到检索出的具体文档（法律名称+条文编号）
2. **可热更新**：法律修订时只需替换知识库文档并重新向量化，无需重训模型
3. **减少幻觉**：模型被限定在给定材料范围内作答，减少虚构法条的概率

### 4.3 LangChain4J 框架核心组件

LangChain4J 是 LangChain 的 Java 移植版，核心抽象包括：

| 接口 | 说明 | 本系统实现 |
|------|------|-----------|
| ChatModel | 对话模型接口 | OpenAiChatModel (指向Qwen-Plus) |
| EmbeddingModel | 向量化模型接口 | OpenAiEmbeddingModel (指向Text-Embedding-v3) |
| EmbeddingStore | 向量存储接口 | RedisEmbeddingStore (RediSearch) |
| ContentRetriever | 内容检索器接口 | QueryRewriteRetriever → HybridContentRetriever |
| ChatMemory | 对话记忆接口 | MessageWindowChatMemory (窗口10轮) |
| DocumentSplitter | 文档切分器 | LegalDocumentSplitter (按"条"切分) |
| AiService | 声明式AI服务 | ConsultantService |

**AiService 声明式编程模型**：开发者只需定义接口方法和注解，框架自动完成提示词注入、检索器调用、记忆管理、流式生成等流程的编排。

## 5. 知识库构建

### 5.1 知识库规模

- **法律法规数量**：189 部
- **涵盖范围**：
  - 国家法律层：民法典、民事诉讼法、刑法、劳动法、公司法、保险法、个人信息保护法等
  - 国务院行政法规：工伤保险条例、医疗事故处理条例、物业管理条例等
  - 最高人民法院司法解释：民法典各编解释、买卖合同纠纷解释、民间借贷规定等
  - 浙江省地方性法规：浙江省物业管理条例、浙江省电动自行车管理条例等
- **文档格式**：.txt（结构化法律）和 .docx（法规条例）

### 5.2 文档加载与解析

```java
// CommonConfig.java 中
Path contentDir = Paths.get("src/main/resources/content");
List<Document> documents = FileSystemDocumentLoader.loadDocumentsRecursively(
        contentDir, new ApacheTikaDocumentParser());
```

使用 Apache Tika 统一解析不同格式的法律文档，支持：
- `.txt`：结构化法律文本（民法典、民事诉讼法等）
- `.docx`：法规条例（工伤保险条例、物业管理条例等）
- `.pdf`：框架支持但当前知识库未使用

### 5.3 文档切分策略（LegalDocumentSplitter）

这是系统的第一个核心技术点。切分策略直接影响检索粒度——切分太粗则检索不精确，切分太细则丢失上下文。

#### 5.3.1 按"第X条"切分（主力策略）

中文法律条文以"第X条"为基本单位。切分器通过正则表达式识别：

```java
// 匹配法条开头：第X条
Pattern.compile("^[　\\s]*第[" + CN_NUMS + "]+条[之的]?[" + CN_NUMS + "]*[　\\s]");
// CN_NUMS = "一二三四五六七八九十百千零○〇0-9"

// 匹配结构标题：第X编/章/节
Pattern.compile("^[　\\s]*(第[" + CN_NUMS + "]+(?:编|分编|章|节))[　\\s]+(.+)");
```

**切分逻辑**：

1. 逐行扫描法律文本
2. 遇到"第X编/章/节"标题时，更新当前章节变量（如"第二章 物权的保护"），但不作为独立片段
3. 遇到"第X条"时，产出上一段的文本片段并附带元数据，然后开始新的一条
4. 不属于"第X条"的连续行（款、项、子项），追加到当前条文
5. 文件结束时保存最后一条

#### 5.3.2 元数据注入

每个切分片段附带三段元数据：

```java
Metadata metadata = new Metadata();
metadata.put("law_name", "中华人民共和国民法典");      // 取自文件名
metadata.put("chapter", "第二章 物权的保护");          // 所在章节
metadata.put("article_number", "第一百四十八条");     // 条文编号
```

同时，每条片段的正文前自动拼接上下文前缀：

```
【中华人民共和国民法典】 第二章 物权的保护
第一百四十八条　一方以欺诈手段，使对方在违背真实意思的情况下实施的民事法律行为，受欺诈方有权请求人民法院或者仲裁机构予以撤销。
```

这样做的好处是：向量化时将法律来源信息嵌入到语义空间中，有利于检索时按法律归属和章节进行匹配。

#### 5.3.3 回退切分

对于没有"第X条"结构的文档（如部分行政法规），按空行分段，每段最大2000字符。

### 5.4 向量化与存储

#### 5.4.1 嵌入模型

使用阿里云 Text-Embedding-v3 模型，将文本映射为 1024 维浮点数向量。关键的语义信息被编码到向量的各个维度中，语义相近的文本在向量空间中距离更近。

#### 5.4.2 Redis 向量存储

```java
RedisEmbeddingStore.builder()
    .host("localhost")
    .port(6379)
    .dimension(1024)                   // 对应 Text-Embedding-v3 输出维度
    .indexName("legal-embeddings")     // 索引名称
    .build();
```

Redis Stack 的 RediSearch 模块支持向量相似度搜索，支持两种索引算法：
- **FLAT**（暴力搜索）：精确但慢，适合数据量不大的场景
- **HNSW**（层次导航小世界图）：近似搜索，快但略牺牲精度

本系统采用 FLAT 算法，因为在万级数据量下性能差异不大（单次检索约200ms）。

#### 5.4.3 批量导入与版本检测

```java
// CommonConfig.java 核心逻辑
String storedVersion = redisTemplate.opsForValue().get("legal:embedding:version");
if (!EMBEDDING_VERSION.equals(storedVersion)) {
    // 版本不一致 → 重新向量化
    embedAndStore(allSegments, redisEmbeddingStore, embeddingModel);
    redisTemplate.opsForValue().set("legal:embedding:version", EMBEDDING_VERSION);
} else {
    log.info("法律文档向量库已是最新版本(v2), 跳过导入");
}
```

- 每批 10 条调用 embedding API
- 全量 189 部法律切分为数千个片段
- 启动时检查 Redis 中的版本标记，避免每次启动都重新计算向量（节省API调用成本）

## 6. 检索管道详解

检索管道是系统的核心技术组件，采用四级流水线设计，以装饰器模式串联。

### 6.1 整体架构

```
QueryRewriteRetriever (装饰器：查询改写 + 合并)
    │
    └── HybridContentRetriever (混合检索器：RRF融合)
            │
            ├── EmbeddingStoreContentRetriever (向量语义检索)
            │       检索范围: Redis RediSearch，Top-20，minScore>0.1
            │
            └── KeywordContentRetriever (关键词精确匹配)
                    检索范围: 全量内存片段，Top-20
```

### 6.2 向量语义检索

#### 6.2.1 原理

向量检索的核心是**将文本映射到高维向量空间，通过计算向量之间的距离（相似度）来衡量语义相关性**。语义相近的文本，其向量在空间中也更接近。

例如：
- 用户问："离婚财产怎么分"
- 向量模型将其映射到与"夫妻共同财产分割"附近的语义空间
- 即使用户提问中完全不含"夫妻"、"分割"等法条原文用词，也能命中相关条文

#### 6.2.2 余弦相似度

Redis RediSearch 支持余弦相似度（Cosine Similarity）度量。两个向量 A 和 B 的余弦相似度计算公式为：

```
cosine(A, B) = (A·B) / (|A| × |B|)
              = Σ(Ai × Bi) / sqrt(Σ(Ai²) × Σ(Bi²))
```

- 取值范围 [-1, 1]，越接近 1 表示越相似
- 相比欧氏距离，余弦相似度不受向量长度影响，更关注方向一致性

#### 6.2.3 配置参数

```java
EmbeddingStoreContentRetriever.builder()
    .embeddingStore(redisEmbeddingStore)
    .embeddingModel(embeddingModel)
    .maxResults(20)       // 返回最多20条
    .minScore(0.1)        // 最低相似度阈值
    .build();
```

- **maxResults = 20**：返回 Top-20 个最相似的文本片段
- **minScore = 0.1**：过滤得分低于 0.1 的结果（经验值，过高的阈值可能漏掉弱相关但有用的结果）

#### 6.2.4 向量检索的优势与局限

**优势**：
- 能捕获语义层面的相似性
- 对同义词、近义表达有较好的泛化能力
- 不依赖精确字符串匹配

**局限**：
- 在精确法条编号匹配上表现不佳（如用户搜"第148条"，向量检索可能因为数字148与内容中其他数字混淆而排序出错）
- 长尾法律术语可能因训练数据不足而映射不准确

### 6.3 关键词精确检索

关键词检索恰好弥补向量检索在精确匹配上的短板。

#### 6.3.1 关键词提取流程

```java
// KeywordContentRetriever.extractKeywords()
1. 去除虚词/语气词：的、了、吗、呢、吧、啊、呀、哦、么、在、是、有、被、把、给、让、跟、和、与、及
2. 按标点切分：，。？！、；：""''（）《》【】和空白字符
3. 保留长度 ≥ 2 的词作为关键词
4. 阿拉伯数字 → 中文数字转换（如检测到"第148条"，额外生成"第一百四十八条"）
```

#### 6.3.2 阿拉伯数字转中文数字算法

这是系统的一个精巧实现。用户习惯用阿拉伯数字（"148条"），而法律原文使用中文数字（"第一百四十八条"）。转换函数支持 1~9999 范围，正确处理了中文数字的特殊规则：

```java
// 关键规则：
// 1. "十五" 而非 "一十五"（十位为1时省略"一"）
// 2. "一百一十五" 而非 "一百十五"（百位后十位非零时省略"一"）
// 3. "二百零五"（百位后十位为零时补"零"）
// 4. "一千二百六十"（连续数位正常拼接）

static String arabicToChinese(int num) {
    if (num <= 0) return "";
    StringBuilder sb = new StringBuilder();
    // 处理千位
    if (num >= 1000) {
        sb.append(CN_DIGITS[num / 1000]).append("千");
        num %= 1000;
        if (num > 0 && num < 100) sb.append("零");  // 如一千零五十
    }
    // 处理百位
    if (num >= 100) {
        sb.append(CN_DIGITS[num / 100]).append("百");
        num %= 100;
        if (num > 0 && num < 10) sb.append("零");   // 如二百零五
    }
    // 处理十位
    if (num >= 10) {
        int tens = num / 10;
        if (sb.isEmpty() && tens == 1) {
            sb.append("十");  // "十五"而非"一十五"
        } else {
            sb.append(CN_DIGITS[tens]).append("十");
        }
        num %= 10;
    }
    // 处理个位
    if (num > 0) {
        sb.append(CN_DIGITS[num]);
    }
    return sb.toString();
}
```

#### 6.3.3 命中率计算

```java
// 对每个文本片段计算命中率
double score = (double) hits / keywords.size();
// hits: 在该片段的text + law_name + chapter中命中的关键词数
```

匹配范围不限于正文，也涵盖法律名称和章节元数据，以提升法律名称关键词的命中概率。

与初期方案不同，最终方案**不设固定得分阈值**——虽然阈值能过滤低质量匹配，但也可能把仅命中部分关键词的正确片段误删。改为纯排序截断后，由下游的 RRF 融合环节统一决定哪些结果值得保留。

#### 6.3.4 为什么不用 BM25 或 TF-IDF？

BM25 依赖词频和文档频率，在短查询场景下（用户问题通常只有一两句话），词频信息有限。此外，BM25 需要预先建立倒排索引，对于长度不一的法律条文片段，工程复杂度较高。本系统的关键词命中率方法更轻量，不需要额外索引，直接在内存中完成扫描。

### 6.4 等权 RRF 融合（HybridContentRetriever）

#### 6.4.1 RRF 算法原理

RRF（Reciprocal Rank Fusion，倒数排名融合）的核心思想是**不直接比较两种异构检索器的得分（向量为余弦相似度，关键词为命中率，量纲不同），而是基于各自列表中的排名位置计算融合得分**。

**标准 RRF 公式**：

```
RRF_score(d) = Σ w_i / (k + rank_i)
```

其中：
- `d`：文档（文本片段）
- `i`：检索器索引
- `w_i`：第 i 个检索器的权重
- `k`：常数，通常取 60（平滑参数，减少排名靠前结果的过度影响）
- `rank_i`：文档 d 在第 i 个检索器结果中的排名（从 1 开始）

#### 6.4.2 为什么取 k=60？

k 的作用是平滑排名差异。假设 k=0，则 rank=1 和 rank=2 的得分比是 2:1；rank=1 和 rank=100 的得分比是 100:1——排名靠前的结果主导性过强。取 k=60 后，rank=1 得分为 1/61≈0.0164，rank=2 为 1/62≈0.0161，差异缩小，排名靠后的结果仍有机会贡献得分。

#### 6.4.3 权重配置的演化（重要！）

**早期方案（不等权）**：

```
加权公式：Score = 1.5/(60 + rank_vector) + 0.5/(60 + rank_keyword)
向量权重 1.5, 关键词权重 0.5
```

**问题发现**：实验验证发现，在此配置下：
- 关键词最高得分 = 0.5/61 ≈ 0.0082
- 向量最低得分 = 1.5/80 ≈ 0.0188
- **关键词所有结果的得分均低于向量任何结果的得分**

即关键词检索的贡献被完全压制，融合实质上退化为"截断版向量检索"。

**最终方案（等权）**：

```java
// 加权公式：Score = 1.0/(60 + rank_vector) + 1.0/(60 + rank_keyword)
HybridContentRetriever(vectorOnlyRetriever, keywordOnlyRetriever, 20, 1.0, 1.0);
```

等权配置下，两个检索器的得分范围完全对称：
- 关键词头部结果（rank=1）：1/61 ≈ 0.0164
- 向量头部结果（rank=1）：1/61 ≈ 0.0164
- **两者真正获得了同台竞争的机会**

#### 6.4.4 去重机制

```java
// 以 "法律名称:条文编号" 作为去重键
String key = law + ":" + article;  // 如 "中华人民共和国民法典:第一百四十八条"
```

同一法条出现在两个检索列表时：
- 得分叠加（协同增强效应）——该条文在两个检索器中都被排到前列，说明高度相关
- 独有结果按各自的排名得分参与公平排序

#### 6.4.5 完整融合流程

```
向量检索结果 (Top-20)          关键词检索结果 (Top-20)
      │                              │
      └────── RRF 等权融合 ──────────┘
                  │
          Score = 1.0/(60+rank_v) + 1.0/(60+rank_kw)
                  │
             去重 key = law_name:article_number
                  │
           按融合得分降序排列
                  │
             截取 Top-20
```

### 6.5 装饰器模式设计

四个检索器以**装饰器模式**（Decorator Pattern）串联：

```
ContentRetriever (接口)
    ├── EmbeddingStoreContentRetriever (LangChain4J 内置)
    ├── KeywordContentRetriever (自定义)
    ├── HybridContentRetriever (组合向量+关键词)
    └── QueryRewriteRetriever (包装 HybridContentRetriever)
```

装饰器模式的优点：
- **单一职责**：每个检索器只负责一种检索逻辑
- **可组合**：可以灵活组合不同的检索器（纯向量、纯关键词、混合、混合+改写）
- **易测试**：每个检索器可独立测试，四组消融实验正是依赖此设计
- **不改动原有代码**：QueryRewriteRetriever 包装 HybridContentRetriever 时，HybridContentRetriever 完全不需要修改

## 7. 查询改写

### 7.1 为什么需要查询改写？

用户的法律咨询问题往往带有强烈的口语色彩：

| 用户原始问题 | 改写后 |
|------------|--------|
| "老婆被老公打了怎么办" | "家庭暴力 反家庭暴力法 人身安全保护令" |
| "在网上买了假货能退一赔三吗" | "欺诈行为 消费者权益保护法 惩罚性赔偿" |
| "小区物业乱收费找谁投诉" | "物业服务人 物业管理条例 收费纠纷" |

这种差距对向量检索来说语义信息足够（向量能捕获"假货"与"欺诈"的语义关联），但对关键词检索来说，如果知识库中使用的是"欺诈行为"而非"假货"，则匹配效果会打折扣。

### 7.2 改写提示词设计

```java
static final String REWRITE_PROMPT =

    "1. 提取核心法律概念和关键词\n" +

    "3. 补充可能相关的法律名称（如\"反家庭暴力法\"\"民法典\"）\n" +
```

### 7.4 原始+改写合并策略

改写后的查询**不会替代**原始查询，而是作为额外的一路检索输入：

```java
// 原始查询检索一次
List<Content> originalResults = delegate.retrieve(query);

// 改写查询检索一次
List<Content> rewrittenResults = delegate.retrieve(Query.from(rewritten));

// 交错合并，原始优先
return mergeResults(originalResults, rewrittenResults);
```

合并策略为**交错插入**（interleaving）：original[0], rewritten[0], original[1], rewritten[1], ...

这样做的优势是：
- 保留原始查询的语义覆盖面（口语化表达也有其独特的语义信息）
- 改写带来的法律术语变体提供补充（覆盖关键词检索的短板）
- 原始结果排在前面，保证基本质量；改写结果排在后面，提供补充信息

## 9. 用户鉴权系统

### 9.1 密码加密：BCrypt

```java
private final BCryptPasswordEncoder passwordEncoder = new BCryptPasswordEncoder();

// 注册时加密存储
user.setPassword(passwordEncoder.encode(password));

// 登录时密文比对
passwordEncoder.matches(password, user.getPassword())
```

BCrypt 的特点：
- **单向加密**：无法从密文反推明文
- **盐值内置**：每次加密自动生成随机盐值，同一密码两次加密结果不同
- **计算慢**：故意设计为计算密集型（可配置 rounds），抵御暴力破解

### 9.2 JWT 令牌

```java
// 签发
public String generateToken(Long userId, String username) {
    return JWT.create()
            .withClaim("userId", userId)
            .withClaim("username", username)
            .withExpiresAt(new Date(System.currentTimeMillis() + expiration)) // 7天
            .sign(Algorithm.HMAC256(secret));
}

// 验证
public DecodedJWT verifyToken(String token) {
    return JWT.require(Algorithm.HMAC256(secret))
            .build()
            .verify(token);
}
```

JWT 的特点：
- **无状态**：服务端不存令牌，只需验证签名
- **自包含**：令牌本身携带用户信息（userId, username）
- **过期控制**：expiration = 604800000ms = 7天

### 11.2 实验设计

| 序号 | 配置 | 说明 |
|------|------|------|
| 1 | 仅向量检索 | 基线 |

### 11.3 测试指标

**召回率（Recall）**：被检索到的相关文档数 / 知识库中所有相关文档数

$$Recall@k = \frac{命中数}{总正向测试数}$$

其中"命中"定义为：人工标注的所有关键词均在 Top-K 返回结果的文本或元数据中出现。

分别在 Top-5、Top-10、Top-15、Top-20 四个截断位置统计分级召回率。

### 11.4 测试结果

| 检索策略 | Recall@5 | Recall@10 | Recall@15 | Recall@20 | 平均耗时 |
|---------|----------|-----------|-----------|-----------|---------|
| 向量检索 | 79.0% | 82.5% | 83.5% | 85.0% | 206ms |
| 混合检索 | 77.5% | 81.0% | 84.0% | 85.5% | 212ms |

### 11.5 结果分析

**结论一**：两种策略的召回率均随截断位置放宽而持续提升，符合"返回更多结果则命中概率更高"的直觉。

**结论二**：混合检索在不同截断位置上存在交叉：
- R@5 和 R@10：混合检索略低于向量检索（-1.5pp）
- R@15 和 R@20：混合检索略高于向量检索（+0.5pp）

揭示的机制：在头部位置（Top-5/10），关键词检索引入的噪声挤占了向量高排名结果；随截断放宽至 Top-15/20，关键词补充的精确匹配文档覆盖了向量检索的遗漏。**等权RRF需要足够的排名深度才能体现互补价值**。

**结论三**：精确查询（22条）两种策略召回率均为 100%，表明对于法条编号查询，两种方式都能有效命中。全部增益和损失均发生在语义查询类型中。

**结论四**：耗时方面，向量检索平均 206ms，混合检索 212ms，差异仅 6ms（来自关键词检索的内存扫描），未引入可感知延迟。

### 11.6 混合检索增益/损失分析

**增益案例（4条）**：集中在法律名称精确匹配场景，如"国有土地使用权出让合同纠纷"、"建设工程质量管理"——关键词检索通过字符串精确匹配命中了向量检索遗漏的文档。

**损失案例（3条）**：集中在短查询场景，如"申请强制执行需要满足什么条件"、"打官司可以申请法律援助吗"——去除虚词后仅剩少量词组，关键词检索排序质量下降，噪声挤占了正确结果的位置。

### 12.3 HybridContentRetriever - RRF 融合核心算法

```java
// RRF 融合算法核心
private List<Content> fuse(List<Content> listA, List<Content> listB) {
    Map<String, RRFEntry> entries = new LinkedHashMap<>();

    // 处理向量检索结果
    for (int i = 0; i < listA.size(); i++) {
        String key = deduplicationKey(listA.get(i));
        entries.computeIfAbsent(key, k -> new RRFEntry(listA.get(i)))
                .addScore(weight / (RRF_K + i + 1));
    }

    // 处理关键词检索结果
    for (int i = 0; i < listB.size(); i++) {
        String key = deduplicationKey(listB.get(i));
        entries.computeIfAbsent(key, k -> new RRFEntry(listB.get(i)))
                .addScore(weight / (RRF_K + i + 1));
    }

    // 按融合得分降序，去重，取Top-N
    return entries.values().stream()
            .sorted(Comparator.comparingDouble(RRFEntry::score).reversed())
            .limit(maxResults)
            .map(RRFEntry::content)
            .collect(Collectors.toList());
}
```

**时间复杂度**：O((m+n) log(m+n))，其中 m、n 分别为向量和关键词检索的结果数（各约20条），排序开销可忽略不计。

## 13. 答辩常见问题及回答

### Q1：为什么选择RAG而不是微调（Fine-tuning）？

**答**：RAG方案在知识更新方面有明显的成本优势。当新法律出台或旧法律修订时，只需替换知识库文档并重新计算向量（几十分钟），而微调方案需要重新准备训练数据、重新训练模型（数小时到数天）。此外，RAG天然支持溯源——每条回答能对应到具体检索到的法条，这对法律场景至关重要。微调模型虽然可能回答更流畅，但其内部知识边界不透明。

### Q2：为什么混合检索在R@5上反而低于纯向量检索？

**答**：这是试验中的一个关键发现。等权RRF融合将向量检索和关键词检索在排名中赋予了同等的权重。在严格截断场景（Top-5），关键词检索引入的部分噪声文档挤占了向量检索的高排名结果位置，导致净损失。但随着截断放宽到Top-15/Top-20，关键词检索补充的精确匹配文档逐渐覆盖了向量检索遗漏的正确答案，收益由负转正。这说明等权RRF融合需要足够的排名深度才能体现互补价值。实际部署时，如果应用场景对Top-5结果质量要求极高，可以考虑动态调整权重。

### Q3：为什么用RRF而不是直接对得分归一化后加权？

**答**：两种检索器的得分量纲不同——向量检索得分是余弦相似度（0到1之间），关键词检索得分是命中率（也是0到1之间）。虽然都是0到1，但它们的分布特征完全不同：向量检索的得分分布通常更集中（0.1-0.9），而关键词检索的得分分布更离散（可能是0, 0.25, 0.5, 0.75, 1.0等离散值）。直接归一化加权需要假设两种得分具有可比性，而RRF基于排名位置而非得分大小进行融合，巧妙地绕开了这个假设。

### Q4：为什么要做等权而非加权RRF？

**答**：早期设计采用了向量权重1.5、关键词权重0.5的不等权配置。但实验发现，在该配置下关键词检索的最高RRF得分（0.5/61≈0.0082）仍低于向量检索的最低得分（1.5/80≈0.0188），即关键词检索的贡献被完全压制，融合退化为"截断版向量检索"。改为等权配置后，两个检索器真正获得了同台竞争的机会，关键词结果才能有效补充向量的盲区。

### Q5：查询改写为什么采用"合并"而非"替换"策略？

**答**：改写后的查询不会替代原始查询，原因是：（1）原始口语化查询有其独特的语义覆盖面，特别是向量检索能捕获口语与专业术语之间的语义关联；（2）改写可能出错或遗漏，保留原始查询作为保险；（3）实验中验证合并策略的召回率高于仅使用改写查询。合并采用交错插入方式，原始结果排在前面保证基本质量，改写结果排在后面提供补充。

### Q6：为什么要自己实现关键词检索器而不是用Elasticsearch？

**答**：（1）部署简化：本系统已经用了Redis，多引入一个Elasticsearch会显著增加运维复杂度；（2）数据规模：189部法律切分为数千个片段，数据量不大，内存扫描的开销仅6ms，完全在接受范围内；（3）可控性：自实现的关键词检索器可以做针对性的优化，如阿拉伯数字→中文数字转换，这在通用检索引擎中需要额外配置。

### Q7：为什么选择Redis作为向量数据库而不是Milvus/Qdrant/Pinecone？

**答**：Redis一专多能——在本系统中同时承担了向量检索（RediSearch）、对话记忆缓存（String/JSON）、版本标记（String）三项职责。相比专门的向量数据库，Redis部署更简单（一个Docker容器），社区更成熟，运维成本更低。在万级数据量下的性能表现（单次向量检索约200ms）完全满足需求。

### Q8：文档切分为什么按"条"而不是按段落或固定长度？

**答**：法律文本的天然语义边界就是"条"。按段落切分可能把一条法条拆成两段（丢失完整语义），也可能把两条法条合并成一段（检索粒度太粗）。按固定长度（如512字符）会切断法条的语义完整性。按"条"切分最符合法律文本的结构特征，也便于在溯源时精确定位到"《XX法》第X条"。

### Q9：BCrypt为什么要设计成计算慢？不怕影响用户体验吗？

**答**：BCrypt计算慢是针对暴力破解的防御措施。密码加密只在注册和登录时执行一次（各约300ms），对用户体验影响极小。但如果密码数据库泄露，攻击者通过彩虹表或暴力猜测密码的成本会大大增加。这是安全领域的标准做法——用可承受的计算开销换取显著提升的安全防护。

### Q10：JWT无状态怎么处理用户注销？

**答**：当前实现中，注销是纯前端操作——清除 localStorage 中的 token。Token 在有效期内技术上仍然可用，这确实是 JWT 无状态设计的一个固有权衡。在生产环境中可以考虑引入黑名单机制（将已注销的 token 存入 Redis，TTL 等于 token 剩余有效期），但这会增加服务端状态。对于学术项目而言，当前方案已足够。

### Q11：系统的检索和生成延迟是多少？

**答**：检索环节平均约212ms（混合检索），生成环节受Qwen-Plus API响应时间影响，通常2-8秒完成一段完整的法律分析。查询改写设置了15秒超时（如果超时则跳过改写直接用原始查询）。总延迟的瓶颈在LLM生成而非检索。

### Q12：系统如何保证不泄露敏感信息（如API Key）？

**答**：当前版本 API Key 直接写在 application.yml 中。论文中已指出这是待改进项——生产环境应使用环境变量或密钥管理服务（如阿里云 KMS）存储。不过作为学术原型系统，当前方式不影响功能验证。

### 14.1 知识库方面

### 14.3 生成方面

- **开源模型**：将开源法律模型做领域微调并本地部署，可能在质量、延迟和成本之间取得更优平衡

## 附录D：RRF 融合算法的数学推导

设向量检索结果列表为 $V = [v_1, v_2, ..., v_m]$，排名分别为 $1, 2, ..., m$。

设关键词检索结果列表为 $K = [k_1, k_2, ..., k_n]$，排名分别为 $1, 2, ..., n$。

对于文档 $d$，其 RRF 融合得分为：

$$RRF(d) = \begin{cases}
\frac{w_v}{k + rank_v(d)} + \frac{w_k}{k + rank_k(d)} & \text{if } d \in V \cap K \\
\frac{w_v}{k + rank_v(d)} & \text{if } d \in V \setminus K \\
\frac{w_k}{k + rank_k(d)} & \text{if } d \in K \setminus V
\end{cases}$$

其中 $k=60$，$w_v=w_k=1.0$（等权配置）。

去重 key 为 $law\_name + ":" + article\_number$，同一条文在两个列表中出现时得分叠加。

最终按 $RRF(d)$ 降序排列，取前 20 条。
