## 1. Recall@k 和 Precision@k 到底怎么算

### 1.1 Recall@k 是评测指标，不是检索算法

Recall@k 用于评价检索结果是否找回目标法律依据。它不是关键词检索内部的打分逻辑，也不是向量检索内部的相似度阈值。

测试数据中每条正向样本有一个 `expect` 或期望命中内容，例如：

```json
{
  "query": "老板不给工资怎么办",
  "expect": "劳动保障监察条例,拖欠工资"
}
```

系统实际检索时只接收 `query` 字段，也就是：

```text
老板不给工资怎么办
```

`expect` 字段不参与检索，只在检索完成后用于判断结果是否命中。

### 1.2 Recall@k 的判定方式

对于某个问题，系统返回 Top-k 检索结果。如果 Top-k 结果中的文本或元数据覆盖该样本的期望命中内容，就认为该问题召回成功。

公式：

```text
Recall@k = N(hit@k) / N(positive)
```

其中：

```text
N(hit@k)：Top-k 内成功命中目标依据的问题数量
N(positive)：正向测试问题总数
```

举例：

```text
正向样本数：360
Top-5 命中的问题数：314
Recall@5 = 314 / 360 = 87.2%
```

所以 Recall@k 是“以问题为单位”的二值统计：

```text
某题 Top-k 命中：1
某题 Top-k 未命中：0
```

然后对全部正向问题求比例。

### 1.3 Recall@k 不是“Top-5 里命中 4 条就是 80%”

如果某个问题的 Top-5 结果中有 4 条相关，那更接近 Precision@5 的概念。

Recall@5 主要看：

```text
Top-5 里有没有覆盖目标依据
```

有则该题 Recall@5 记为 1，没有则记为 0。

### 1.4 Precision@k 的计算方式

Precision@k 用于衡量 Top-k 结果中相关条目所占比例。

对于单个问题：

```text
Precision@k = Top-k 中相关条目数量 / k
```

对全部正向问题取平均：

```text
Precision@k = 1 / N_positive × Σ(R_i@k / k)
```

其中：

```text
R_i@k：第 i 个问题的 Top-k 结果中相关条目数量
```

举例：

```text
某问题 Top-5 结果中有 4 条可以支持回答
Precision@5 = 4 / 5 = 80%
```

### 1.5 是否用“余弦相似度 > 0.1”判断准确召回

不是。

`minScore = 0.1` 是向量检索器内部的候选过滤阈值，用于决定哪些向量结果可以返回。它不等于评测中的“准确召回”。

准确召回的判断依据仍然是：

```text
Top-k 检索结果是否覆盖测试样本的期望命中内容
```

也就是说：

```text
向量相似度 > 0.1：只是允许进入候选结果
覆盖 expect：才算评测命中
```

## 2. 向量检索的原理和代码位置

### 2.1 向量检索做什么

向量检索用于捕捉语义相似性。它不要求用户问题和法律条文使用完全相同的词。

例如：

```text
用户问题：离婚财产怎么分
法律表述：夫妻共同财产分割
```

即使问题中没有“夫妻共同财产分割”这些原文词，向量模型仍可能把二者映射到相近的语义空间位置。

### 2.2 向量怎么生成

代码位置：

```text
src/main/java/org/example/consultant3/aiservice/config/CommonConfig.java
```

关键配置：

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiEmbeddingModel.builder()
            .baseUrl(dashscopeBaseUrl)
            .apiKey(dashscopeApiKey)
            .modelName("text-embedding-v3")
            .build();
}
```

系统使用阿里云 DashScope 的 `text-embedding-v3` 模型，将法律文本片段和用户问题转换为 1024 维向量。

Redis 向量库配置：

```java
@Bean
public RedisEmbeddingStore redisEmbeddingStore() {
    return RedisEmbeddingStore.builder()
            .host(redisHost)
            .port(redisPort)
            .dimension(1024)
            .indexName("legal-embeddings")
            .build();
}
```

### 2.3 向量检索怎么排名

向量检索按照用户问题向量和法律文本片段向量之间的相似度排序。论文中可以解释为余弦相似度：

```text
cos(q, d) = (q · d) / (||q|| × ||d||)
```

其中：

```text
q：用户问题向量
d：法律文本片段向量
q · d：两个向量的点积
||q||、||d||：向量模长
```

余弦相似度越高，说明问题和法律片段语义越接近，排名越靠前。

### 2.4 当前向量检索参数

代码位置：

```text
CommonConfig.java
```

配置：

```java
this.vectorOnlyRetriever = EmbeddingStoreContentRetriever.builder()
        .embeddingStore(redisEmbeddingStore)
        .embeddingModel(embeddingModel)
        .maxResults(20)
        .minScore(0.1)
        .build();
```

含义：

```text
maxResults = 20：最多返回 20 条
minScore = 0.1：只返回相似度分数大于 0.1 的结果
```

答辩回答：

> 向量检索会把用户问题转成 1024 维查询向量，然后在 Redis RediSearch 的 legal-embeddings 向量索引中检索相似度最高的法律片段。系统最多返回 20 条，且设置了 0.1 的最低相似度过滤阈值。

## 3. 关键词检索的原理和代码位置

### 3.1 关键词检索做什么

关键词检索是词面匹配，不理解语义。它主要看用户问题中的关键词是否出现在法律文本、法律名称或章节里。

它适合：

```text
法律名称明确：民法典、劳动合同法、消费者权益保护法
条文编号明确：第148条、第76条
法律术语明确：工伤认定、劳动仲裁、物业服务合同
```

不擅长：

```text
老板不给工资怎么办
买到假货怎么办
老公打我怎么办
```

因为法律文本中可能写的是：

```text
用人单位拖欠劳动报酬
经营者欺诈行为
家庭暴力
```

当前关键词检索不会自动理解这些同义关系。

### 3.2 关键词检索代码位置

```text
src/main/java/org/example/consultant3/aiservice/config/KeywordContentRetriever.java
```

### 3.3 关键词怎么提取

核心方法：

```java
List<String> extractKeywords(String query)
```

流程：

1. 去除常见虚词、语气词。

```java
String cleaned = query.replaceAll("[的了吗呢吧啊呀哦么在是有被把给让跟和与及]", " ");
```

会去掉：

```text
的、了、吗、呢、吧、啊、在、是、有、被、把、给、让、和、与、及
```

2. 按标点和空格切分。

```java
cleaned = cleaned.replaceAll("[，。？！、；：“”‘’（）《》【】\\s]+", " ").trim();
```

3. 保留长度不少于 2 的片段。

```java
for (String part : cleaned.split("\\s+")) {
    if (part.length() >= 2) {
        keywords.add(part);
    }
}
```

4. 识别阿拉伯数字法条编号，并转换成中文数字条号。

```java
private static final Pattern ARABIC_ARTICLE = Pattern.compile("第(\\d+)条");
```

例如：

```text
第148条 → 第一百四十八条
第76条 → 第七十六条
```

### 3.4 关键词匹配范围

代码：

```java
String lawName = seg.metadata().getString("law_name");
String chapter = seg.metadata().getString("chapter");
String combined = seg.text()
        + (lawName != null ? " " + lawName : "")
        + (chapter != null ? " " + chapter : "");
```

关键词不是只匹配正文，还匹配：

```text
法条正文
法律名称 law_name
章节 chapter
```

### 3.5 关键词 score 怎么算

代码：

```java
private double score(String text, List<String> keywords) {
    int hits = 0;
    for (String kw : keywords) {
        if (text.contains(kw)) hits++;
    }
    return (double) hits / keywords.size();
}
```

公式：

```text
关键词得分 = 命中关键词数量 / 查询关键词总数
```

注意：这个分母是“查询中提取出的关键词数量”，不是法条中的全部关键词数量。

举例：

```text
查询关键词：民法典、第一百四十八条、欺诈
某法条命中：民法典、第一百四十八条
关键词得分 = 2 / 3 = 0.667
```

答辩回答：

> 关键词检索不是把问题关键词和法条所有关键词做比值，而是统计查询关键词在某个文本片段中命中了多少。分母是查询关键词总数，分子是该片段命中的查询关键词数量。

### 3.6 关键词检索为什么效果低

主要原因：

1. 当前实现是规则式切分，不是真正中文分词。
2. 没有法律术语词典。
3. 没有同义词扩展。
4. 没有 BM25、TF-IDF 等权重算法。
5. 测试集多为口语化自然语言问题，更适合向量检索。

答辩回答：

> 关键词检索单独效果较低，是因为当前关键词检索主要基于字符串包含匹配，难以处理口语表达与法律术语之间的差异。例如“老板不给工资”在法律文本中常写作“用人单位拖欠劳动报酬”，简单关键词匹配难以直接命中。后续可以引入 BM25、中文分词和法律术语词典提升关键词检索效果。

## 4. 混合检索的原理和当前代码实现

### 4.1 混合检索做什么

混合检索把向量检索和关键词检索的结果进行融合。

目的：

```text
向量检索负责语义匹配
关键词检索负责法律名称、条号、明确术语等精确匹配
```

### 4.2 混合检索代码位置

```text
src/main/java/org/example/consultant3/aiservice/config/HybridContentRetriever.java
```

### 4.3 当前代码的重要提醒

当前 `HybridContentRetriever.java` 已经不只是最早的等权 RRF，它现在包含了：

1. 查询类型识别；
2. 关键词结果准入过滤；
3. 动态权重；
4. RRF 融合；
5. 按法律名称 + 条文编号去重。

如果论文还写“纯等权 RRF”，要注意和代码当前状态可能不完全一致。答辩时按论文还是按代码，需要口径统一。

### 4.4 混合检索流程

代码核心：

```java
public List<Content> retrieve(Query query) {
    List<Content> vectorResults = vectorRetriever.retrieve(query);
    List<Content> keywordResults = keywordRetriever.retrieve(query);

    String queryText = query.text();
    QueryType queryType = classify(queryText);
    List<Content> admittedKeywordResults = filterKeywordResults(queryText, keywordResults);
    RetrievalWeights weights = weightsFor(queryType, admittedKeywordResults);

    return fuse(vectorResults, admittedKeywordResults, weights);
}
```

流程：

```text
用户问题
  ↓
向量检索得到 Top20
  ↓
关键词检索得到 Top20
  ↓
识别查询类型
  ↓
过滤低质量关键词结果
  ↓
根据查询类型设置向量/关键词权重
  ↓
RRF 融合
  ↓
按 law_name + article_number 去重
  ↓
返回 Top20
```

### 4.5 查询类型识别

代码：

```java
private QueryType classify(String queryText) {
    if (ARTICLE_PATTERN.matcher(queryText).find() || LAW_NAME_PATTERN.matcher(queryText).find() || queryText.contains("《")) {
        return QueryType.EXACT;
    }
    if (LEGAL_TERMS.stream().anyMatch(queryText::contains)) {
        return QueryType.LEGAL_TERM;
    }
    return QueryType.COLLOQUIAL;
}
```

分三类：

```text
EXACT：包含法条编号、法律名称、书名号等精确线索
LEGAL_TERM：包含工伤、劳动仲裁、合同无效等法律术语
COLLOQUIAL：普通口语化表达
```

### 4.6 动态权重

代码：

```java
private RetrievalWeights weightsFor(QueryType queryType, List<Content> admittedKeywordResults) {
    if (admittedKeywordResults.isEmpty()) {
        return new RetrievalWeights(vectorWeight, 0.0);
    }
    return switch (queryType) {
        case EXACT -> new RetrievalWeights(1.0, 1.5);
        case LEGAL_TERM -> new RetrievalWeights(1.0, 0.8);
        case COLLOQUIAL -> new RetrievalWeights(1.0, 0.1);
    };
}
```

含义：

```text
精确查询：关键词权重更高
法律术语查询：关键词有一定权重
口语化查询：关键词权重很低，避免噪声干扰向量结果
如果关键词结果全部被过滤：关键词权重为 0
```

### 4.7 关键词结果准入过滤

代码：

```java
private boolean isAdmittedKeywordResult(String queryText, Content content, List<String> effectiveKeywords) {
    ...
    if (lawName != null && queryText.contains(lawName)) return true;
    if (articleNumber != null && queryText.contains(articleNumber.replaceAll("[^零一二三四五六七八九十百千万\\d]", ""))) return true;
    if (LEGAL_TERMS.stream().anyMatch(term -> queryText.contains(term) && text.contains(term))) return true;

    long hits = effectiveKeywords.stream().filter(text::contains).count();
    if (hits >= 2) return true;
    return !effectiveKeywords.isEmpty() && (double) hits / effectiveKeywords.size() >= KEYWORD_SCORE_THRESHOLD;
}
```

准入条件包括：

```text
命中法律名称
命中条文编号
命中核心法律术语
有效关键词命中数 >= 2
关键词命中率 >= 0.4
```

### 4.8 RRF 融合公式

代码：

```java
.addScore(weights.vector() / (RRF_K + i + 1));
.addScore(weights.keyword() / (RRF_K + i + 1));
```

其中：

```java
private static final int RRF_K = 60;
```

公式：

```text
Score(d) = wv / (60 + rank_vector(d)) + wk / (60 + rank_keyword(d))
```

其中：

```text
wv：向量检索权重
wk：关键词检索权重
rank_vector(d)：文档 d 在向量结果中的排名
rank_keyword(d)：文档 d 在关键词结果中的排名
```

如果某条法条只出现在向量结果中，只计算向量部分。如果同时出现在向量和关键词结果中，两个分数会叠加，排名会更靠前。

### 4.9 混合检索去重条件

代码：

```java
private String deduplicationKey(Content content) {
    TextSegment seg = content.textSegment();
    String law = seg.metadata().getString("law_name");
    String article = seg.metadata().getString("article_number");
    if (law != null && article != null) {
        return law + ":" + article;
    }
    String text = seg.text();
    return text.substring(0, Math.min(100, text.length()));
}
```

去重条件不是“法条 + 关键词”。

真正去重键是：

```text
law_name + article_number
```

也就是：

```text
法律名称 + 条文编号
```

例如：

```text
中华人民共和国民法典:第一百四十八条
```

如果元数据缺失，则用文本前 100 个字符作为兜底 key。

答辩回答：

> 混合检索的去重依据是法律名称和条文编号，而不是关键词。这样可以避免同一条法条同时被向量检索和关键词检索召回时重复展示。如果没有条文编号，则使用文本前 100 个字符作为兜底去重依据。

## 5. 查询改写检索器

### 5.1 代码位置

```text
src/main/java/org/example/consultant3/aiservice/config/QueryRewriteRetriever.java
```

### 5.2 做什么

查询改写用于把用户口语化问题转换成更适合法律检索的表达。

例如：

```text
老公打我怎么办
```

可能改写为：

```text
家庭暴力 反家庭暴力法 人身安全保护令 民法典
```

### 5.3 当前代码流程

```text
原始问题
  ↓
调用 Qwen-Plus 改写
  ↓
原始查询执行一次混合检索
  ↓
改写查询执行一次混合检索
  ↓
交错合并
  ↓
按 law_name + article_number 去重
  ↓
返回 Top20
```

注意：当前代码不是“原始结果全部排前面，再追加改写结果”。它是交错合并：

```text
原始第1条
改写第1条
原始第2条
改写第2条
...
```

这样可以保证改写查询的结果有机会进入最终结果。

```java
private void embedAndStore(List<TextSegment> segments,
                           RedisEmbeddingStore store,
                           EmbeddingModel embeddingModel) {
    int batchSize = 10;
    for (int i = 0; i < segments.size(); i += batchSize) {
        List<TextSegment> batch = segments.subList(i, Math.min(i + batchSize, segments.size()));
        List<Embedding> embeddings = embeddingModel.embedAll(batch).content();
        store.addAll(embeddings, batch);
    }
}
```

系统每批 10 条文本片段生成 embedding，然后写入 Redis。

## 12. 常见答辩问题与回答

### Q1：Recall@k 是不是匹配关键词就算成功？

答：Recall@k 是评测指标。系统检索时只输入问题文本，不输入答案字段。检索完成后，测试脚本根据样本的期望命中内容判断 Top-k 结果是否覆盖目标法律名称、条文编号或核心关键词。如果覆盖，就认为该问题在 k 范围内召回成功。

### Q2：向量检索的准确召回是不是看相似度大于 0.1？

答：不是。0.1 是向量检索器的候选过滤阈值，用于过滤相似度过低的结果。实验中的召回成功仍然要看 Top-k 结果是否覆盖测试样本的期望命中内容。

### Q3：关键词检索多少关键词命中算有效召回？

答：关键词检索内部先用命中率排序，命中率是“命中查询关键词数 / 查询关键词总数”。但实验中的有效召回不是由关键词检索内部阈值决定，而是由评测脚本判断 Top-k 结果是否覆盖该样本的期望命中内容。

### Q4：关键词检索 score 的分母是什么？

答：分母是用户查询中提取出的关键词数量，不是法条中的全部关键词数量。

### Q5：混合检索怎么融合？

答：先分别得到向量检索结果和关键词检索结果，再使用 RRF 根据排名位置计算融合得分。当前代码还会根据查询类型动态调整权重，对关键词结果设置准入过滤，最后按融合分数排序返回 Top20。

### Q6：混合检索怎么去重？

答：用法律名称 `law_name` 和条文编号 `article_number` 作为去重 key。同一条法条如果同时出现在向量和关键词结果中，只保留一条，但融合分数会叠加。如果元数据缺失，则用文本前 100 个字符兜底去重。

### Q7：为什么关键词检索效果低？

答：当前关键词检索是规则式词面匹配，没有中文分词、BM25、法律术语词典和同义词扩展。测试集中口语化问题较多，例如“老板不给工资”对应法律文本中的“用人单位拖欠劳动报酬”，简单关键词匹配难以覆盖，因此效果低于向量检索。

### Q8：为什么还保留关键词检索？

答：关键词检索虽然整体召回低，但在法律名称、条文编号、明确法律术语查询中有价值，响应速度也很快。它适合作为向量检索的低成本精确补充。

### Q9：为什么需要查询改写？

答：查询改写用于把口语表达转成法律术语表达。它不能保证所有查询都提升，但在短查询、口语化表达、法律术语缺失的问题上有补充价值。

### Q10：法条溯源是从模型回答里解析出来的吗？

答：不是。法条溯源来自检索结果的元数据。系统通过 `/chat/sources` 接口重新调用检索管道，返回法律名称、章节、条文编号和摘要。这样比从模型回答中解析引用更稳定。

### Q11：游客模式和登录模式区别？

答：游客可以直接问答，对话保存在浏览器 localStorage；登录用户携带 JWT，服务端会把会话和消息保存到 MySQL，支持历史会话查看和跨设备访问。

### Q12：系统主要代码结构怎么讲？

答：Controller 层负责接口，aiservice/config 包负责 RAG 检索管道和模型配置，aiservice/ConsultantService 负责 LangChain4J 声明式 AI 服务，entity/mapper 负责 MySQL 表和数据访问，repository/RedisChatMemoryStore 负责 Redis 对话记忆，resources/content 保存法律文档，resources/static/index.html 是前端页面。
