# 验收速查：项目目录与代码接口对应

本文档用于项目验收时快速回答“功能在哪个文件实现”“接口对应哪些代码”“前端在哪里调用接口”“核心算法在哪看”等问题。

## 1. 项目整体说明

本项目是一个基于 Spring Boot + LangChain4J + Redis 向量库 + MySQL 的法律智能问答系统。

核心流程：

```text
用户在前端输入问题
  ↓
/chat 接口接收问题
  ↓
ConsultantService 调用大模型
  ↓
contentRetriever 执行 RAG 检索
  ↓
QueryRewriteRetriever 查询改写
  ↓
HybridContentRetriever 混合检索融合
  ↓
向量检索 + 关键词检索 + 等权 RRF
  ↓
大模型基于检索内容生成流式回答
  ↓
前端展示回答和引用法条
```

## 5. 接口总表：前端调用到后端代码

### 5.1 用户接口

| 接口 | 方法 | 前端调用位置 | Controller | Service | Mapper/表 | 说明 |
|---|---|---|---|---|---|---|
| `/api/user/register` | POST | `index.html:664` | `UserController.java:19` | `UserService.java:25` | `UserMapper` → `user` 表 | 用户注册，密码 BCrypt 加密，注册成功返回 JWT |


## 9. 核心算法位置

### 9.1 向量检索

文件：`CommonConfig.java`

关键代码：

```java
this.vectorOnlyRetriever = EmbeddingStoreContentRetriever.builder()
        .embeddingStore(redisEmbeddingStore)
        .embeddingModel(embeddingModel)
        .maxResults(20)
        .minScore(0.1)
        .build();
```

验收说法：向量检索把用户问题转成 embedding，再到 Redis 向量库中找相似法律片段，最多返回 20 条，相似度低于 0.1 的会过滤。

### 9.2 关键词检索

文件：`KeywordContentRetriever.java`

关键逻辑：

- `extractKeywords()`：去除常见虚词，按标点和空格切分关键词。
- `arabicToChinese()`：把 `第148条` 补充为 `第一百四十八条`，提升法条编号匹配能力。
- `score()`：计算命中关键词数 / 总关键词数。
- `retrieve()`：按分数降序返回 TopN。

验收说法：关键词检索解决精确词、法律名称、法条编号匹配的问题，和向量语义检索互补。

### 9.3 等权 RRF 混合融合

文件：`HybridContentRetriever.java`

关键位置：

- `RRF_K = 60`：RRF 平滑常数。
- `fuse()`：融合两路结果。
- `weightsFor()`：当前有关键词结果时返回 `1.0 / 1.0` 等权。
- `deduplicationKey()`：用 `law_name:article_number` 去重。

当前融合公式：

```text
score(d) = vector_score + keyword_score
vector_score = 1.0 / (60 + rank_vector)
keyword_score = 1.0 / (60 + rank_keyword)
```

代码中因为循环下标从 0 开始，实际计算是：

```java
1.0 / (RRF_K + i + 1)
```

验收说法：系统不直接比较向量相似度和关键词分数，因为两者量纲不同；RRF 只看排名位置。当前已经改成等权 RRF，向量和关键词各占 1.0。

### 9.4 查询改写

文件：`QueryRewriteRetriever.java`

关键逻辑：

- `rewrite()`：调用大模型把口语化问题改写成法律检索查询。
- `retrieve()`：先用原始问题检索，再用改写问题检索。
- `mergeResults()`：原始查询和改写查询结果交错合并，并按法条去重。

验收说法：查询改写不是替换原始问题，而是“原始查询 + 改写查询”一起检索，避免丢失原问题中的精确信息。

```text
统计 Recall@5/10/15、Precision@5/10/15/20 等指标
```

## 12. 验收常问速答

### Q1：前端页面在哪？

答：`src/main/resources/static/index.html`。这是一个 Vue 3 静态页面，页面结构、样式和接口请求都在这个文件里。

### Q2：聊天接口在哪？

答：后端入口是 `ChatController.java:36` 的 `/chat`，它调用 `ConsultantService.java:23`，由 LangChain4J 连接大模型、记忆和 RAG 检索器。

### Q3：登录接口在哪？

答：`UserController.java:19` 是注册接口，`UserController.java:28` 是登录接口；具体业务逻辑在 `UserService.java`。

### Q4：历史会话接口在哪？

答：`ConversationController.java`。`/list` 查会话列表，`/create` 新建会话，`/{id}` 删除会话，`/{id}/messages` 查消息。

### Q5：JWT 在哪里校验？

答：`JwtInterceptor.java` 负责校验 token，`WebConfig.java` 把这个拦截器注册到所有请求。

### Q6：哪些接口不需要登录？

答：注册、登录不需要；`/chat`、`/chat/sources`、`/test/` 是可选登录；`/api/conversation/**` 必须登录。

### Q7：法律文档在哪里？

答：原始文档在 `src/main/resources/content/`，启动时由 `CommonConfig.java` 加载，再交给 `LegalDocumentSplitter.java` 切分成法条片段。

### Q8：向量库在哪里？

答：Redis 中的 `legal-embeddings` 索引。配置在 `CommonConfig.java` 的 `redisEmbeddingStore()`。

### Q9：RRF 算法在哪里？

答：`HybridContentRetriever.java`。`fuse()` 方法做 RRF 融合，`weightsFor()` 当前返回等权 `1.0 / 1.0`。

### Q10：关键词检索在哪里？

答：`KeywordContentRetriever.java`。它负责关键词提取、条号数字转中文、关键词命中率打分。

### Q11：查询改写在哪里？

答：`QueryRewriteRetriever.java`。它用 LLM 把口语问题改写为法律检索查询，再把原始查询和改写查询的结果合并。

### Q12：系统提示词在哪里？

答：`src/main/resources/system.txt`，`ConsultantService.java:21` 通过 `@SystemMessage(fromResource = "system.txt")` 引入。

### Q13：测试接口在哪里？

答：`TestController.java`，包括 `/test/compare`、`/test/precision`、`/test/segments`、`/test/performance`、`/test/batch`。

### Q14：为什么要混合检索？

答：向量检索擅长语义相似，关键词检索擅长精确命中法律名称和法条编号。混合检索通过 RRF 按排名融合两者，避免直接比较不同量纲的原始分数。

### Q15：为什么要查询改写？

答：用户问题往往是口语表达，例如“老公打我怎么办”，查询改写会补充“家庭暴力”“反家庭暴力法”等法律术语，提高检索召回。

### Q16：登录和未登录有什么区别？

答：登录后会话和消息存 MySQL，可以跨浏览器恢复；未登录时只保存在浏览器 localStorage。

### Q17：回答内容怎么保存？

答：`ChatController.java` 里，用户问题在调用大模型前写入 `message` 表，助手回答在流式输出完成后通过 `doOnComplete` 写入 `message` 表。

### Q18：法条引用是否就是模型回答里的内容？

答：不是。引用法条是前端在回答结束后额外调用 `/chat/sources`，后端用检索器重新查到的相关法条元数据，用于展示来源。

## 15. 一句话总结

本项目前端在 `static/index.html`，接口入口集中在 `controller` 包，用户和会话数据走 MySQL，法律知识库走 Redis 向量库，核心 RAG 检索链路在 `aiservice/config` 包，其中 `HybridContentRetriever.java` 实现向量检索和关键词检索的等权 RRF 融合。
