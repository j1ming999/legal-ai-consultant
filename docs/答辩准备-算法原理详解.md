# 答辩准备：系统算法与原理详解

## 一、系统整体架构

用户问题 → 查询改写(LLM) → 混合检索(向量+关键词+RRF融合) → 检索结果 → LLM生成回答(带系统提示词+对话记忆)

完整检索管道的调用链：
```
QueryRewriteRetriever
  └── HybridContentRetriever (等权RRF融合)
        ├── EmbeddingStoreContentRetriever (向量语义检索, Redis HNSW)
        └── KeywordContentRetriever (关键词命中率检索, 内存)
```

## 二、知识库构建

### 2.1 文档加载
- 使用 Apache Tika 解析器，支持 .txt / .docx / .pdf
- 从配置目录递归加载所有法律文件
- 法律名称 = 文件名去掉扩展名（如 `中华人民共和国民法典.txt` → `中华人民共和国民法典`）

### 2.2 法条切分算法（LegalDocumentSplitter）

**核心思路：** 以"第X条"为切分边界，每条法律条文作为一个独立的文本片段。

**正则表达式：**
- 法条匹配：`^[　\s]*第[一二三四五六七八九十百千零○〇0-9]+条[之的]?[一二三四五六七八九十百千零○〇0-9]*[　\s]`
- 结构标题匹配：`^[　\s]*(第[...]+(?:编|分编|章|节))[　\s]+(.+)`
- 条号提取：`第[...]+条[之的]?[...]*`

**处理流程：**
1. 逐行扫描文档
2. 遇到结构标题（编/章/节）→ 更新当前章节上下文变量，不生成片段
3. 遇到"第X条"开头 → 保存上一条，开始新的一条
4. 其他行 → 追加到当前条文（属于款、项、子项等）
5. 文档结束 → 保存最后一条

**上下文注入：** 每个片段的文本前自动拼接 `【法律名称】 章节名\n原文`
- 目的：让向量化时法律来源信息被编码进语义空间，提升按法律归属匹配的能力
- 例：`【中华人民共和国民法典】 第三编 合同 第一分编 通则 第八章 违约责任\n第五百七十七条 当事人一方不履行合同义务...`

**元数据：** 每个片段附带三个字段
- `law_name`：法律名称
- `chapter`：所在章节
- `article_number`：条文编号（如"第一百四十八条"）

**兜底方案：** 如果文档中没有检测到任何"第X条"格式（如部分行政法规），按空行分段，每段最大2000字符。

### 2.3 向量化与存储

- 模型：阿里云 DashScope text-embedding-v3，输出维度 1024
- 存储：Redis Stack RediSearch 模块，HNSW 索引，余弦相似度
- 索引名：`legal-embeddings`
- 批量处理：每批 10 条片段调用 embedding API
- 版本控制：Redis key `legal:embedding:version` 存储当前版本号（如"v2"），启动时比对，一致则跳过重新向量化

**答辩可能问的问题：**
- Q: 为什么选 Redis 而不是 Milvus/Weaviate？
- A: 知识库规模数万条，Redis 单实例即可承载；同时 Redis 还承担对话记忆缓存、版本标记等职责，减少系统组件数量；部署简单运维成本低。

- Q: 为什么用 HNSW 而不是暴力搜索？
- A: HNSW 在高维空间（1024维）中检索速度远优于暴力搜索，且精度损失极小。多层跳表式图结构从顶层稀疏图向底层稠密图逐层导航，平衡了速度和精度。

- Q: 为什么 minScore 设为 0.1？
- A: 余弦相似度范围 [-1, 1]，0.1 是一个很宽松的阈值，目的是过滤完全不相关的结果（如随机文本），同时不误杀弱相关但可能有用的条文。实际有效结果的相似度通常在 0.3-0.8 之间。

## 三、检索管道算法

### 3.1 向量语义检索

**原理：** 将用户查询和知识库条文都映射到同一个 1024 维向量空间，通过余弦相似度衡量语义相关性。

**余弦相似度公式：**
```
cos(A, B) = (A · B) / (||A|| × ||B||)
```
其中 A 是查询向量，B 是文档向量。值域 [-1, 1]，越接近 1 越相似。

**配置：** maxResults=20, minScore=0.1

**优势：** 能捕捉跨表达的语义匹配。如"离婚财产怎么分"能匹配到"夫妻共同财产分割"相关条文。

**劣势：** 对精确的法条编号匹配能力弱。如"第148条"可能与其他数字混淆。

### 3.2 关键词检索算法（KeywordContentRetriever）

**步骤一：关键词提取**
1. 去除中文虚词/语气词：`的、了、吗、呢、吧、啊、呀、哦、么、在、是、有、被、把、给、让、跟、和、与、及`
2. 按标点符号和空格切分：`，。？！、；：""''（）《》【】` + 空白字符
3. 保留长度 ≥ 2 的词组作为关键词
4. 阿拉伯数字条号转换：检测"第\d+条"模式，转为中文数字作为额外关键词

**步骤二：阿拉伯数字→中文数字转换**

支持 1~9999，规则：
- 千位：`num/1000` + "千"，余数 < 100 则补"零"
- 百位：`num/100` + "百"，余数 < 10 则补"零"
- 十位：最高位为十位且为1时省略"一"（如 15→"十五" 而非"一十五"）
- 个位：直接映射

示例：
- 148 → "一百四十八"
- 205 → "二百零五"
- 15 → "十五"
- 1260 → "一千二百六十"

**步骤三：命中率打分**
```
score = 命中的关键词数 / 总关键词数
```
对每个文本片段，将 `正文 + law_name + chapter` 拼接为搜索范围，逐个检查关键词是否出现（contains 精确匹配）。

**步骤四：** 按得分降序排列，返回 Top-20。不设固定阈值，由下游 RRF 融合决定取舍。

**答辩可能问的问题：**
- Q: 为什么不用 BM25？
- A: BM25 需要建立倒排索引和 IDF 统计，适合大规模文档集。本系统知识库规模有限（数万条），且法律条文的关键匹配场景主要是法律名称和条文编号的精确匹配，二值命中计数足够且更可解释。延迟仅 8ms。

- Q: 为什么不设阈值？
- A: 早期设过阈值，但发现会误杀仅命中部分关键词的正确片段。改为纯排序截断后，由 RRF 融合统一决定哪些结果值得保留。

### 3.3 RRF 融合算法（HybridContentRetriever）

**RRF（Reciprocal Rank Fusion）原理：**

不直接比较两种检索器的原始分数（向量用余弦相似度，关键词用命中率，量纲不同），而是基于排名位置计算融合分数。

**公式：**
```
Score(d) = vectorWeight / (k + rank_vector + 1) + keywordWeight / (k + rank_keyword + 1)
```
- k = 60（RRF 原论文推荐值，平滑排名靠后结果的得分差异）
- rank 从 0 开始（第1名 rank=0）
- vectorWeight = 1.0, keywordWeight = 1.0（等权配置）

**计算示例：**
假设某条文在向量检索中排第1（rank=0），在关键词检索中排第3（rank=2）：
```
Score = 1.0/(60+0+1) + 1.0/(60+2+1) = 1/61 + 1/63 = 0.01639 + 0.01587 = 0.03226
```

假设某条文只出现在向量检索中排第1：
```
Score = 1/61 + 0 = 0.01639
```

假设某条文同时出现在两个列表中（协同增强）：得分叠加，排名提升。

**去重逻辑：** 以 `law_name:article_number` 为去重键。同一条文出现在两个列表中时，RRF 得分叠加（协同增强效应）。

**为什么选等权而非不等权？**
早期用过向量1.5、关键词0.5的配置。问题是：关键词结果的最高 RRF 得分 = 0.5/(60+0+1) ≈ 0.0082，低于向量结果的最低得分 1.5/(60+19+1) ≈ 0.0188，关键词完全无法参与排序。等权配置下两路得分范围对称，关键词头部结果能与向量结果公平竞争。

**答辩可能问的问题：**
- Q: k=60 是怎么来的？
- A: RRF 原论文（Cormack et al., 2009）通过实验确定 k=60 在多数场景下表现最优。k 越大，排名靠后的结果得分差异越小（长尾更平滑）；k 越小，头部结果优势越明显。60 是一个经验值。

- Q: 为什么混合检索的 Precision 比纯向量低？
- A: 关键词检索在语义查询为主的测试集上精确率仅 24.4%，它引入的低相关性结果通过 RRF 进入最终排序，挤占了头部位置。这是经典的 Precision-Recall 权衡：混合检索牺牲精确率换取更高的召回覆盖面。

- Q: 怎么改进？
- A: 可以降低关键词权重（如 0.5）来抑制噪声；或对关键词检索设置最低得分阈值过滤低质量结果；或根据查询类型自适应调整权重（精确查询提高关键词权重，语义查询降低）。

### 3.4 查询改写算法（QueryRewriteRetriever）

**改写提示词：**
```
你是一个法律检索查询优化器。请将用户的口语化问题改写为专业的法律检索查询。

改写规则：
1. 提取核心法律概念和关键词
2. 将口语表达替换为法律术语（如"老公打我"→"家庭暴力"）
3. 补充可能相关的法律名称（如"反家庭暴力法""民法典"）
4. 如果用户提到具体法条编号，保留原样
5. 只输出改写后的查询，不要解释

用户问题：{原始查询}
改写后的检索查询：
```

**安全机制：**
- 超时保护：LLM 调用设置 15 秒超时，超时则使用原始查询
- 异常兜底：任何异常都 fallback 到原始查询
- 长度校验：改写结果超过原始查询 3 倍长度则丢弃（防止模型输出过长无关内容）
- 空结果校验：改写为空则使用原始查询

**合并策略：交错合并（Interleaved Merge）**
```
结果 = [primary[0], secondary[0], primary[1], secondary[1], ...]
```
- primary = 原始查询的混合检索结果
- secondary = 改写查询的混合检索结果
- 按 `law_name:article_number` 去重（先出现的保留）
- 截断到 Top-20

**为什么用交错合并而非简单拼接？**
简单拼接（原始结果在前，改写结果在后）会导致改写结果只能填充原始结果去重后的空位。交错合并让改写查询的头部结果有机会进入最终列表的前半部分，更公平地传递改写带来的排序增益。

**答辩可能问的问题：**
- Q: 改写模型为什么也用 qwen-plus？
- A: 改写任务需要理解法律术语映射关系，需要足够的语言能力。用同一个模型减少了额外的 API 配置和成本。

- Q: 改写带来了多少延迟？
- A: 平均 2640ms - 234ms ≈ 2.4 秒额外延迟，主要是 LLM API 调用。这是当前系统响应速度的主要瓶颈。

- Q: 如何优化延迟？
- A: 可以对相同或相似查询缓存改写结果；或使用更轻量的模型（如 qwen-turbo）做改写；或将改写与原始查询的检索并行执行。

## 四、对话生成

### 4.1 系统提示词约束
```
核心规范：
1. 必须引用具体法条，格式：根据《XX法》第X条
2. 先梳理法律关系，再逐一分析
3. 涉及多法律领域时分别引用
4. 不暴露 RAG 机制（不说"根据检索到的资料"）
5. 结尾加免责声明
6. 只回答法律问题，其他礼貌拒绝
7. 不编造不存在的法条
```

### 4.2 对话记忆
- 实现：MessageWindowChatMemory，滑动窗口保留最近 10 条消息（5轮对话）
- 存储：RedisChatMemoryStore，JSON 序列化，键名 `user-{userId}-conv-{convId}`，TTL 1天
- 作用：让模型理解上下文，支持追问

### 4.3 流式输出
- 接口返回 `Flux<String>`（Spring WebFlux 响应式流）
- 传输方式：Transfer-Encoding: chunked
- 前端：Fetch API + ReadableStream 逐字节读取追加显示

## 五、评估指标算法

### 5.1 Recall@k（召回率）

**定义：** 在 Top-k 检索结果中，是否覆盖了所有标注的期望关键词。

**计算方法：**
1. 对 Top-k 中每条结果，拼接 `law_name + chapter + article_number + text` 为一个字符串
2. 逐个检查期望关键词是否出现在结果集的任意一条中（跨文档覆盖）
3. 如果所有期望关键词都被覆盖 → 该用例命中（hit=1）
4. Recall@k = 命中用例数 / 正向测试总数(195)

**关键设计决策：** 允许关键词分布在多个结果中（跨文档覆盖），而非要求所有关键词出现在同一条结果中。原因：用户查询的关键词可能分布在多个相关法条片段中。

### 5.2 Precision@k（精确率）

**定义：** Top-k 检索结果中，与问题相关的条目占比。

**计算方法：**
1. 对 Top-k 中每条结果，拼接 `law_name + chapter + article_number + text`
2. 如果该条结果包含至少一个期望关键词 → 视为相关（relevant）
3. Precision@k = 相关条目数 / k
4. 最终输出所有正向测试用例的平均 Precision@k

**与 Recall 的区别：**
- Recall 问的是"正确答案有没有被找到"（以用例为单位，二值判定）
- Precision 问的是"返回的结果中有多少是有用的"（以单条结果为单位，比例）

### 5.3 负面测试评估

对于 `expect = "NOT_IN_KB"` 的测试用例：
- 判定标准：检索器返回 ≤ 5 条结果视为通过
- 目的：验证系统对知识库外问题不会过度检索（不会返回大量不相关结果误导 LLM）

### 5.4 实验结果核心数据

**消融实验（表6.1）：**

| 策略 | R@5 | R@20 | P@5 | P@20 | 耗时 |
|------|-----|------|-----|------|------|
| 仅向量 | 81.0% | 87.2% | 91.1% | 84.5% | 191ms |
| 仅关键词 | 13.8% | 17.9% | 24.4% | 22.5% | 8ms |
| 混合RRF | 79.5% | 87.7% | 65.7% | 56.2% | 234ms |
| 完整管道 | 81.5% | 88.2% | 69.1% | 61.6% | 2640ms |

**核心结论：**
1. 向量检索是召回能力核心（87.2%），且精确率最高（P@5=91.1%）
2. 混合检索以精确率下降（91.1%→65.7%）为代价换取召回率微升（87.2%→87.7%）——经典 Precision-Recall 权衡
3. 查询改写在两项指标上均优于混合检索（R@20: 88.2%, P@5: 69.1%）

**查询改写细分（表6.2）：**

| 路径 | R@5 | R@10 | R@15 | R@20 |
|------|-----|------|------|------|
| 原始→混合 | 79.5% | 83.1% | 86.2% | 87.7% |
| 改写→混合 | 80.5% | 83.6% | 85.1% | 87.2% |
| 合并 | 81.5% | 86.2% | 86.2% | 88.2% |

**核心结论：**
- 改写查询头部排序更优（R@5: +1.0pp），但召回广度略低（R@20: -0.5pp）
- 合并后 R@5=81.5% 超过两路各自表现，证明互补有效

## 六、答辩高频问题预判

### Q: 为什么不直接微调一个法律大模型？
A: 微调需要大量标注数据和 GPU 资源，且每次法律更新都要重新训练。RAG 方案只需替换知识库文档即可热更新，成本低、可维护性强。

### Q: 向量检索已经 87.2% 了，为什么还要加关键词检索和改写？
A: 向量检索在精确法条编号匹配上有盲区（如"第148条"可能与其他数字混淆）。关键词检索在 4 条向量检索无法覆盖的精确匹配场景提供了关键补充，且额外开销仅 8ms。查询改写进一步将 R@20 从 87.7% 提升至 88.2%，P@5 从 65.7% 提升至 69.1%。

### Q: 混合检索精确率下降这么多（91%→66%），值得吗？
A: 在 RAG 系统中，检索结果是喂给 LLM 的上下文，LLM 有能力从噪声中筛选有用信息。因此召回率（不漏掉正确答案）比精确率（结果纯净度）更重要。但精确率过低会增加 LLM 的负担和幻觉风险，这是后续优化方向。

### Q: 测试集 200 条够吗？
A: 对于本科毕设的概念验证足够。200 条覆盖了精确查询(22条)、语义查询(173条)、负面测试(5条)三种类型，涵盖了民法典、消费者权益保护法、道路交通安全法等多部法律。更大规模的评估是后续工作。

### Q: Precision 的判定标准是否太宽松（只要包含一个关键词就算相关）？
A: 确实是一个简化的判定。更严格的做法是人工标注每条结果的相关性等级（如 0/1/2 分），计算 nDCG。但对于本系统的评估目的（对比不同策略的相对表现），当前标准已经能区分出策略间的差异。

### Q: 查询改写失败了怎么办？
A: 三重保护：15秒超时自动降级、异常捕获降级、长度异常降级。所有降级都 fallback 到原始查询直接检索，不影响系统可用性。

### Q: 为什么对话记忆只保留 5 轮？
A: LLM 的上下文窗口有限，保留过多历史会挤占检索结果的空间。5 轮（10条消息）足以支持常见的追问场景，同时为检索结果留出足够的 token 空间。

### Q: 系统怎么处理知识库外的问题？
A: 两层防护：(1) 检索层——如果问题与知识库无关，向量相似度低，返回结果少或相关性差；(2) 生成层——系统提示词要求"不编造不存在的法条"、"如实告知并建议咨询专业律师"。负面测试显示 5 条知识库外问题均未返回高置信度结果。

### Q: 为什么用 Redis 存对话记忆而不是 MySQL？
A: 对话记忆是高频读写的临时数据（每次对话都要读取和更新），Redis 的内存读写速度远优于 MySQL 的磁盘 IO。设置 1 天 TTL 自动过期，避免无限增长。持久化的对话记录（用于历史查看）仍存在 MySQL 中。

### Q: RRF 中 rank 从 0 开始还是从 1 开始？
A: 代码中 rank 从 0 开始（循环变量 i 从 0 开始），所以公式是 `weight / (k + i + 1)`。第1名的得分 = weight/(60+0+1) = weight/61。这与论文中 `1/(k+rank)` 的表述等价（论文中 rank 从 1 开始）。

### Q: 文本切分时"第X条之一"怎么处理？
A: 正则表达式中包含了 `[之的]?[中文数字]*` 的可选后缀，能匹配"第二十六条之一"、"第一百四十八条之二"等格式。
